# 수입 부대비용 안분 자동화 — 기획·기술설계 통합 설계서

> **문서 성격**: 기획(PRD) + 기술설계 통합 / 개발 착수 가능 수준
> **구현 스택**: Next.js 15 (App Router) · Supabase · shadcn/ui · Claude API (Vision/언어모델)
> **작성 근거**: `docs/plandoc.pdf`(기획 슬라이드 10p), `docs/최종정산서_XMCCLINC26050074.pdf`(15p), `docs/입고현황.xlsx`(205행), `docs/검증로직.md`
> **버전**: v1.0 (초안)

---

## 1. 개요

### 1.1 한 줄 정의
영문 **수입신고필증**과 한글 **입고현황(이카운트 양식)**을 AI로 자동 매칭하고, **수입정산서의 부대비용을 품목별로 안분**하며, **입고일 기준 환율**을 자동 적용해 **이카운트 업로드 양식을 자동 완성**하는 웹 애플리케이션.

### 1.2 배경 (AS-IS — 현재는 수작업)
현재 업무 흐름은 다음과 같이 전부 수작업이다.

```
영업팀이 양식에 입력해 회신 → 담당자가 신고필증과 눈으로 대조 → 환율·부대비용 손작업 → 입력
   (영업팀 회신)        (수기 대조)              (부대비용 안분)        (환율·결과 기입)
```

**무엇이 비효율인가**

| 문제 | 내용 |
|------|------|
| 언어가 다름 | 신고필증은 **영문**, 입고현황은 **한글** — 같은 품목인지 사람이 일일이 판단 |
| 건당 반복 소모 | 매 수입 건마다 품목 대조·수량 확인·안분 계산을 처음부터 반복 |
| 오류·누락 위험 | 수량 오타와 안분 계산 실수가 원가·재고 자료에 그대로 반영 |

### 1.3 목표 (TO-BE)
> 영업팀은 양식 회신만, 사람은 애매한 건 확인만 — 매칭·안분·환율·기입은 자동.

```
영업팀 회신 → 매칭·BL확정 → 안분·환율 → 자동 기입 → 확인·확정
 (영업팀)       (AI)        (AI)       (AI)       (사람)
            └──────── AI 자동 처리 구간: 사람 손 없이 수행 ────────┘
```

### 1.4 기대 효과
- **시간 절감**: 건당 수기 대조·계산·조회 시간 제거 (사람은 애매 매칭 확정만 수행)
- **정확도 향상**: 수량 합계 검증(㉟=㊶), 안분 합계 = 정산서 총액 일치 검증으로 오류 차단
- **추적성**: 매칭 근거·안분 기준·환율 출처를 데이터로 남겨 사후 검증 가능
- **표준화**: 이카운트 업로드 양식의 컬럼·순서를 그대로 유지해 ERP 업로드까지 무중단 연결

## 2. 용어 정의

| 용어 | 정의 |
|------|------|
| **수입정산서** | 포워더(예: MYUNGIL LOGISTICS)가 발행하는 정산 문서. 부대비용 항목·과세/비과세·환율·B/L번호·관세율을 담음. 최종정산서 PDF의 **1페이지** |
| **세금계산서** | 정산서에 첨부되는 세금계산서. **본 시스템에서는 처리 제외** (PDF 2~3p) |
| **수입신고필증** | 관세청(UNI-PASS) 발급 영문 신고 문서. 품목·신고수량·외화단가·과세가격·세액을 담음. 스캔 이미지(PDF 4p~). **Claude Vision OCR 대상** |
| **이카운트(ECOUNT)** | 영업팀이 사용하는 ERP. 입고현황 업로드 양식을 제공. 최종 결과물은 이 양식을 그대로 채운 형태 |
| **부대비용** | 수입에 부수되는 비용(해상운임·THC·통관수수료·관세·부가세 등). 정산서에서 추출해 품목별로 **안분**함 |
| **안분(按分)** | 총액을 품목별 수량 비중 등 기준으로 나눠 배분하는 것 |
| **BL번호 (B/L No.)** | 선하증권 번호. 입고현황에는 없고, 매칭 확정 시 신고필증·정산서로부터 자동으로 채워짐 |
| **㉟수량 / ㊶수량** | 신고필증의 35번(품목별 신고수량) / 41번(환급물량 합계) 항목. 검증의 핵심 키 |
| **통화코드** | 이카운트 통화 코드. **USD = 00001, CNY = 00002** |

## 3. 입력 / 출력 명세

### 3.1 입력 1 — 최종정산서 PDF (여러 장, 동적 구조)
하나의 PDF에 여러 문서가 합쳐져 있으며, **페이지 수는 건마다 가변**이다. 시스템은 페이지 종류를 동적으로 분류한다.

| 페이지 | 문서 종류 | 처리 방식 |
|--------|-----------|-----------|
| 1 | 수입정산서 | **텍스트 추출** (PDF 텍스트 레이어 존재) → 부대비용·환율·B/L·관세율 파싱 |
| 2 ~ 3 | 세금계산서 | **무시 (처리 제외)** |
| 4 ~ N | 수입신고필증 | **스캔 이미지** (텍스트 레이어 0자) → **Claude Vision OCR** |
| N+1 ~ | ACE 보험증권 등 | **무시** (검증 범위 종료 신호) |

**신고필증 경계 판정 규칙** (`검증로직.md` 준수)
- **4페이지부터**, `"ACE American Fire and Marine Insurance Company Korea, A Chubb Company"` 문구가 나타나기 **직전 페이지까지**를 수입신고필증으로 간주한다.
- 실측 사례(XMCCLINC26050074): 4~7p가 신고필증, **8페이지부터 ACE 보험증권** 시작 → 신고필증 = 4~7p.
- 페이지 종류는 "텍스트 레이어 유무 + 키워드(세금계산서/ACE/UNI-PASS·수입신고필증)"로 동적 판정한다.

**수입정산서(1p)에서 추출하는 핵심 값** (실측)

| 번호 | 필드명 | 실측 예시 |

### 3.2 입력 2 — 입고현황 xlsx (1개 파일)
- 시트 예: "5월", **205행** / 이카운트 양식 24컬럼.
- 컬럼 순서(고정):

```
일자 · 순번 · 거래처코드 · 거래처명 · 담당자 · 입고창고 · 통화 · 환율 · 프로젝트 ·
품목코드 · 품목명 · 규격명 · 수량 · 단가 · 외화금액 · 공급가액 · 부가세 · 적요 ·
부대비용 · 운반비 · 보관비 · 수수료 · 기타
```

| 구분 | 컬럼 | 비고 |
|------|------|------|
| **영업팀 입력** | 품목코드, 품목명, 수량, 외화단가, 입고일자 | 회신 시 채워옴 |
| **자동 기입 대상** | 통화(코드), 환율, 단가(원화), 외화금액, 공급가액, 부가세, 운반비, 수수료, 기타(+부대비용/보관비) | 시스템이 계산·기입 |

- 품목코드 예: `BA07012159GY240` (앞 10자리 `BA07012159` = 모델, 뒤 = 색상/사이즈) — **색상·사이즈별로 행 분리**됨.
- 통화코드: USD = `00001`, CNY = `00002`.

## 4. 사용자 시나리오 & 화면 흐름

### 4.1 주 사용자
- **회계 담당자**(주 사용자): PDF·xlsx 업로드, 애매 매칭 확정, 결과 다운로드.
- **영업팀**(간접): 입고현황 양식에 품목·수량·외화단가 입력 후 회신(시스템 외부).

### 4.2 화면 흐름

```
[1] 업로드            [2] 자동 처리(진행률)        [3] 매칭 검토         [4] 안분·환율 결과      [5] 다운로드
 PDF(정산서) +    →   PDF 분류·OCR·정산서 파싱  →  일치/불일치/확인요  →  부대비용 안분 +    →   이카운트 양식 +
 xlsx(입고현황)       수량검증(㉟=㊶)             애매건 1클릭 확정       입고일 환율 자동       검증 리포트
```

| 단계 | 화면 | 주요 동작 |
|------|------|-----------|
| 1 | 업로드 | 최종정산서 PDF(다중 페이지) + 입고현황 xlsx 드래그&드롭. 형식·필수 컬럼 검증 |
| 2 | 처리 진행 | 페이지 분류 → 신고필증 Vision OCR → 정산서 파싱 → ㉟/㊶ 수량 합계 검증. 진행률·로그 표시 |
| 3 | 매칭 검토 | 영문 품목 ↔ 한글 품목 매칭 테이블. 상태 배지(일치/불일치/확인요). **확인요** 행은 후보 점수순 제시 → 클릭 확정 |
| 4 | 안분·환율 | 부대비용 안분 결과 + 입고일 환율 미리보기. 안분 합계=정산서 총액 검증 배지 |
| 5 | 결과 | 이카운트 양식(24컬럼) 미리보기 + 엑셀 다운로드 + 검증 리포트 다운로드 |

## 5. 시스템 아키텍처

### 5.1 구성 (현재 레포 스택)

```
┌──────────────────────────── 브라우저 (Client Components) ────────────────────────────┐
│  업로드 UI · 매칭 검토 테이블 · 결과 미리보기 (shadcn/ui)                               │
└───────────────────────────────────────┬──────────────────────────────────────────────┘
                                         │  (파일 업로드 / 매칭 확정 / 다운로드)
┌────────────────────── Next.js App Router (Server / Route Handlers) ───────────────────┐
│  /api/ingest      PDF 페이지 분류 + 정산서 텍스트 파싱 + xlsx 파싱                       │
│  /api/ocr         수입신고필증 이미지 → Claude Vision OCR (구조화 추출)                  │
│  /api/match       영문↔한글 3단계 매칭 (코드 → Claude 의미매칭 → 후보)                  │
│  /api/fx          입고일 기준 환율 조회 (하나은행/서울외국환중개) + 캐시                  │
│  /api/allocate    부대비용 안분 + 수량 역산 + 검증                                       │
│  /api/export      이카운트 양식 엑셀 생성                                                │
└──────────┬───────────────────────────┬──────────────────────────────┬─────────────────┘
           │                           │                              │
   ┌───────▼────────┐        ┌─────────▼─────────┐          ┌─────────▼──────────┐
   │  Supabase      │        │   Claude API      │          │  환율 소스(외부)    │
   │  Postgres+RLS  │        │  Vision / Text    │          │  하나은행/서울외환  │
   │  Storage(파일) │        │  (OCR·의미매칭)   │          │  (조회+캐시)        │
   └────────────────┘        └───────────────────┘          └────────────────────┘
```

### 5.2 역할 분담
- **Client Components**: 업로드, 매칭 검토 인터랙션, 결과 미리보기. Supabase Client(`lib/supabase/client.ts`).
- **Route Handlers (Server)**: 무거운 처리(OCR·매칭·안분·환율·엑셀). Supabase Server(`lib/supabase/server.ts`)로 매 요청마다 새 클라이언트 생성.
- **Supabase**: 데이터 영속화(배치/추출/매칭/안분/검증 로그) + 원본 파일 Storage + 환율 캐시 + 인증/RLS.
- **Claude API**: ① 신고필증 이미지 Vision OCR(구조화 JSON), ② 영문↔한글 품목명 의미 매칭.

> **CLAUDE.md 준수**: Server Components/Route Handlers에서는 전역 변수 금지·매 호출 시 `createClient()` 신규 생성. 미들웨어의 `getClaims()` 흐름 변경 금지.

## 6. 처리 파이프라인 상세

```
PDF 업로드
  │
  ├─[P1] 페이지 분류  ── 1p:정산서 / 2~3p:세금계산서(skip) / 4p~ACE직전:신고필증 / ACE~:skip
  │
  ├─[P2] 정산서 파싱(1p, 텍스트) ── 부대비용 항목·SUB TOTAL·과세/비과세·관세율·환율·B/L
  │
  ├─[P3] 신고필증 OCR(4p~, 이미지) ── Claude Vision → 라인별 {모델·㉟수량·㊱단가·㊲금액}, 문서별 ㊶수량
  │
  └─[P4] 수량 검증 ── Σ㉟  ==  Σ㊶ ?  (불일치 시 오류 메시지 반환, 진행 차단/경고)

xlsx 업로드
  └─[I1] 입고현황 파싱 ── 행별 {품목코드·품목명·수량·외화단가·입고일자·통화}

매칭·계산
  ├─[M1] 3단계 매칭     ── 코드 앞10자리 → Claude 의미매칭 → 후보(사람 확정)
  ├─[M2] 수량 비교      ── 신고수량 vs 입고수량 → 일치/불일치/확인
  ├─[M3] 외화단가 역산  ── 행별 수량비중으로 단가 배분, Σ(수량×단가)=신고필증 품목총액 검증
  ├─[M4] 환율 적용      ── 입고일 기준 자동 조회(통화코드 자동), 단가/공급가액/부가세 환산
  ├─[M5] 부대비용 안분  ── A)신고수량 기준 or B)입고수량 기준, 칸별 합계=정산서 총액 검증
  └─[M6] 양식 생성      ── 이카운트 24컬럼 채움 + 검증 리포트
```

## 7. 핵심 알고리즘 / 수식

### 7.1 3단계 품목 매칭 (공통 코드 없음)
매칭이 확정되면 그 건의 **BL번호가 자동으로 따라온다.**

| 순위 | 방법 | 근거 |
|------|------|------|
| **1순위** | **품목코드 앞 10자리 ↔ 신고필증 모델번호** 일치 | 입고현황 `BA07012159GY240`의 앞 10자리 `BA07012159` = 신고필증 모델 → 가장 정확. 정규화(공백/대소문자 제거) 후 비교 |
| **2순위** | **영문↔한글 품목명 의미 매칭** | 코드가 안 맞을 때 사전 + Claude 언어모델로 의미 연결 (예: `BALLOP CALORRING DIET ... SLIPPERS` ↔ `밸롭 칼로링 다이어트 지압 슬리퍼`) |
| **3순위** | **사람 확인 후보 제시** | 애매한 건은 매칭 후보를 **점수순**으로 제시 → 담당자 클릭 한 번으로 확정 |

**매칭 상태 정의**

| 상태 | 의미 | 처리 |
|------|------|------|
| 🟢 일치 | 신고수량 = 입고수량 | 신고수량 기준으로 안분 |
| 🔴 불일치 | 신고수량 ≠ 입고수량 | 입고수량 기준 안분 + 차이 표시 |
| 🟡 확인요 | 매칭 애매 | 후보 제시, 사람이 1클릭 확정 |

### 7.2 외화단가 입력 & 수량 역산 보정
같은 품목(모델)이라도 색상·사이즈·발주 시점에 따라 외화단가가 다를 수 있다. 신고필증에는 품목 **합계 금액**만 있어 행마다 단가를 그대로 넣으면 총액이 안 맞는다. → **수량으로 역산해 행별 단가를 보정**, 합계가 신고필증과 일치하게 맞춘다.

```
총 외화금액 = Σ(행별 수량 × 외화단가)

① 신고필증의 품목 총 외화금액을 기준으로 잡는다
② 행별 수량 비중으로 단가를 배분·역산
③ Σ(단가 × 수량) == 총액  → 자동 검증 (불일치 시 플래그 + 재확인 요청)
```

**예시 — 단가가 다른 동일 품목** ('부스트 숏팬츠')

| 품목/옵션 | 수량 | 외화단가 | 외화금액 |
|---|---|---|---|
| 부스트 숏팬츠 BLK | 1,610 | 7.20 | 11,592.0 |
| 부스트 숏팬츠 NVY | 330 | 6.80 | 2,244.0 |
| **합계 (신고필증 일치)** | **1,940** | (역산 보정) | **13,836.0** |

### 7.3 부대비용 안분 규칙
수입정산서의 부대비용을 **매칭된 품목 수량 비율**로 나눠 입고현황에 채운다.

**A) 수량 일치 → 신고수량 기준**
```
품목 안분액 = 부대비용 총액 × (품목수량 ÷ 수량합계)
```
각 부대비용 항목을 같은 비율로 배분 후 운반비·수수료·기타 칸에 분류 기입.

**B) 수량 불일치 → 입고수량 기준**
```
안분액 = 부대비용 총액 × (품목 입고수량 ÷ 입고수량 합계)
```
신고수량이 아니라 **실제 입고수량**으로 나눈다. 해당 품목에 **"입고수량 기준 안분"** 표시 + 차이 비고 기록.

**정산서 항목 → 이카운트 칸 매핑** (각 칸에 들어간 뒤 품목 수량 비율로 안분)

| 이카운트 칸 | 정산서 항목 | 실측 예시 |
|---|---|---|
| **운반비** | 통관수수료 + 업무운임 SUB TOTAL | 50,000 + 2,758,088 원 |
| **수수료** | 통관부가세 | 12,831,620 원 |
| **기타** | 관세 | 0 원 (관세율 0%) |

> ✅ **칸별 안분액 합계 = 정산서 총액** 일치 검증.

### 7.4 원화 환산 수식 (이카운트 칸)
```
단가(원화)  = 외화단가 × 환율
공급가액    = 수량 × 단가(원화) / 1.1        (= 수량 × 외화단가 × 환율 / 1.1)
부가세      = 공급가액 × 10%
통화코드    = USD → 00001, CNY → 00002
```

### 7.5 검증식 (요약)
| 검증 | 식 | 실패 시 |
|---|---|---|
| 신고수량 합계 | `Σ㉟ == Σ㊶` | "수입신고필증의 35번 수량 합계와 41번 수량 합계가 일치하지 않습니다." |
| 단가 역산 | `Σ(수량×외화단가) == 신고필증 품목 총액` | 플래그 + 재확인 요청 |
| 부대비용 안분 | `Σ(품목 안분액) == 정산서 부대비용 총액` | 잔차를 최대 품목에 보정(rounding) 또는 오류 표시 |

## 8. AI(Claude) 활용 설계

### 8.1 수입신고필증 Vision OCR
- **입력**: PDF 신고필증 페이지를 이미지로 렌더링(고해상도, 2x 이상) 후 Claude Vision에 전달.
- **방식**: 구조화 출력(JSON 스키마 강제) — 페이지별로 라인 아이템 배열과 문서 합계를 추출.
- **추출 스키마(요지)**:

```jsonc
{
  "declaration_no": "12085-26-050308M",      // ① 신고번호
  "bl_no": "XMCCLINC26050074",               // ⑤ B/L번호
  "page_index": 4,
  "is_continuation": true,                    // "을지/계속" 여부
  "line_items": [
    { "seq": "001", "model": "BS02010161 TIVAT 2.0",
      "qty_35": 332, "unit_price_usd_36": 12.7, "amount_usd_37": 4216.4 }
  ],
  "qty_41_total": 8594,                        // ㊶ 환급물량(수량)
  "fx_rate_65": 1482.04,
  "duty_rate": 9.10,
  "confidence": 0.97                           // 페이지 OCR 신뢰도
}
```

- **신뢰도 임계값**: `confidence < 0.85` 이거나 숫자 합계 검증 실패 시 **"OCR 저신뢰" 플래그** → 사람 검수 큐로.
- **모델**: 최신 Claude(예: Opus/Sonnet 4.x 계열) 사용. 비용·속도에 따라 OCR=고정밀, 매칭=경량으로 분리 가능.

### 8.2 영문↔한글 품목명 의미 매칭
- 1순위(코드)에서 미매칭된 건만 Claude에 위임.
- **프롬프트 요지**: "다음 영문 신고품목과 한글 입고품목 후보 목록을 의미적으로 매칭하고, 각 후보에 0~1 신뢰도 점수를 부여하라. 브랜드/제품군/기능 키워드를 근거로 설명하라."
- **출력 스키마(요지)**:

```jsonc
{
  "english_item": "BALLOP CALORRING DIET ... SLIPPERS",
  "candidates": [
    { "inventory_code": "BA07012159GY240", "korean_name": "밸롭 칼로링 다이어트 지압 슬리퍼",
      "score": 0.96, "reason": "BALLOP=밸롭, SLIPPERS=슬리퍼, CALORRING DIET=칼로링 다이어트" }
  ]
}
```

- **자동 확정 임계값**: 최고 점수 `≥ 0.90` 이고 2위와 격차 충분 → 자동 확정. 그 외 → "확인요"로 후보 제시.
- **사전(dictionary) 병행**: 자주 나오는 브랜드/제품군 매핑 사전을 Supabase에 적재해 LLM 호출 전 1차 필터 + 학습 자산화(확정 이력을 사전에 누적).

## 9. 환율 연동 설계 (자동 연동 필수)

### 9.1 요구
입고일자를 읽어 **하나은행 '기간별 환율조회'의 최초 고시(매매기준율)**을 자동 적용. 통화코드 자동(USD/CNY). 1차 소스 실패 시 **서울외국환중개소 기간별 매매 기준 환율**로 폴백.

### 9.2 조회 모듈 설계
```
getFxRate(date, currency)  // currency ∈ {USD, CNY}
  1) fx_rate_cache 조회 (date, currency) → hit이면 반환
  2) miss → 1차: 하나은행 기간별 환율조회 (최초 고시)
            실패 → 2차: 서울외국환중개소 기간별 매매기준율
  3) 결과를 fx_rate_cache에 저장(출처·조회시각 포함) 후 반환
  4) 두 소스 모두 실패 → 오류 + 담당자 수동 입력 오버라이드(안전장치)
```

- **재시도/타임아웃**: 각 소스 N회 재시도 + 지수 백오프. 영업일이 아닌 경우 직전 영업일 고시 적용 규칙 명시.
- **캐시**: 동일 (일자, 통화) 재조회 방지 → 성능·외부 의존 최소화.
- **수동 오버라이드**: 자동 연동이 원칙이나, 외부 소스 장애 시 업무 중단을 막기 위한 운영 안전장치로 수동 입력값을 허용(입력 시 "수동" 출처 표기·로그).
- **출처 기록**: 모든 적용 환율에 `source`(hana/smbs/manual), `fetched_at`, `quote_date`를 남겨 추적성 확보.

> 외부 사이트 직접 연동은 공식 무료 API 부재로 **서버 측 조회(Route Handler/Edge Function)에서 수행 + 캐시**하는 형태로 설계한다. 소스별 파서는 어댑터로 분리해 한쪽 변경에 강건하게 한다.

## 12. 검증 · 예외 처리

| 케이스 | 처리 |
|---|---|
| **㉟ ≠ ㊶** | 진행 차단 또는 경고. 메시지: "수입신고필증의 35번 수량 합계와 41번 수량 합계가 일치하지 않습니다." |
| **단가 역산 불일치** | 행별 단가 보정 후에도 총액 불일치 시 플래그 + 재확인 요청 |
| **안분 합계 ≠ 정산서 총액** | 반올림 잔차는 최대 수량 품목에 가산(원 단위) / 큰 차이는 오류 표시 |
| **매칭 미확정(확인요)** | 결과 생성 차단, 해당 행 하이라이트 → 사람 확정 후 진행 |
| **OCR 저신뢰** | `confidence` 낮음/숫자 검증 실패 시 검수 큐, 원본 이미지 병기 |
| **환율 조회 실패** | 1차→2차 폴백, 모두 실패 시 수동 입력 오버라이드(출처 "manual" 기록) |
| **PDF 구조 이상** | ACE 문구 미검출/세금계산서만 존재 등 비정상 → 분류 결과 사용자 확인 단계 제공 |
| **xlsx 컬럼 누락** | 필수 컬럼(품목코드·품목명·수량·외화단가·입고일자) 검증, 누락 시 업로드 거부 |

## 13. 보안·권한 / 비기능 요구 / 로드맵

### 13.1 보안·권한
- Supabase Auth(미들웨어 보호) — `/protected` 하위에 본 기능 배치, 미인증 시 `/auth/login` 리다이렉트.
- 원본 문서·결과는 RLS + 비공개 Storage. AI 호출 시 민감정보 최소 전송, 키는 서버 환경변수.

### 13.2 비기능 요구
| 항목 | 목표 |
|---|---|
| 정확도 | 코드 매칭 100% 자동, 의미 매칭 자동 확정 임계 0.90↑, 검증식 3종 전부 통과 시에만 출력 |
| 성능 | 1배치(정산서 + 신고필증 수 페이지 + 입고현황 수백 행) 처리 수십 초 내 목표(OCR 병렬화) |
| 추적성 | 매칭 근거·안분 기준·환율 출처를 모두 기록 |
| 견고성 | 외부 소스(환율) 어댑터 분리 + 캐시 + 폴백 |

### 13.3 로드맵
- **Phase 1 (MVP)**: 업로드 → 정산서 파싱 + 신고필증 OCR + ㉟/㊶ 검증 → 코드 매칭 → 환율 자동(USD/CNY) → 안분 → 이카운트 양식 다운로드.
- **Phase 2**: AI 의미 매칭 + 후보 1클릭 확정 UX, 매칭 사전 학습 자산화, 단가 역산 보정 고도화.
- **Phase 3**: 배치 이력/대시보드, 환율 소스 이중화·모니터링, 이카운트 직접 연동(가능 시), 권한·감사 로그 강화.
