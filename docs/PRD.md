## 1. 핵심 정보

**목적**: 영문 수입신고필증(PDF 스캔)과 한글 입고현황(xlsx)을 AI로 자동 매칭하고 부대비용을 품목별 안분하여 이카운트 업로드 양식을 자동 완성함으로써, 회계 담당자의 수작업 대조·계산·기입을 제거한다.

**사용자**: 수입 상품을 이카운트 ERP로 관리하는 중소 무역·유통사의 회계 담당자(1~3인 소규모 팀, 매월 수~수십 건의 수입 정산을 처리).

## 2. 문제 정의

### AS-IS (현재 수작업 흐름)

```
영업팀 입고현황 회신
  → 담당자가 영문 신고필증과 한글 입고현황을 눈으로 대조
  → 부대비용 수기 안분 (운반비/수수료/기타 칸)
  → 입고일 환율 수기 조회 (하나은행 사이트)
  → 이카운트 양식에 수기 기입
```

| 문제 | 영향 |
|------|------|
| 언어 불일치: 신고필증은 영문, 입고현황은 한글 | 같은 품목인지 사람이 매번 판단, 실수 발생 |
| 건당 반복 소모 | 매 수입 건마다 처음부터 수행, 시간 손실 |
| 수량 오타·안분 계산 실수 | 원가·재고 데이터에 그대로 반영 |
| 추적성 없음 | 어떤 근거로 안분했는지 사후 검증 불가 |

### TO-BE (이번 시스템)

```
파일 업로드 (PDF + xlsx)
  → AI 자동 처리 (분류·OCR·파싱·검증)
  → 매칭 검토 (코드 자동 + 애매건 1클릭 확정)
  → 안분·환율 자동 계산
  → 이카운트 양식 다운로드
```

## 3. 사용자 여정

```
1. 로그인 페이지
   ↓ 인증 완료

2. 배치 목록 페이지 (홈)
   ↓ "새 정산 처리" 버튼 클릭

3. 업로드 페이지
   ↓ PDF + xlsx 드래그&드롭 → "처리 시작" 버튼

4. 처리 진행 페이지 (자동)
   ↓ PDF 분류 → 정산서 파싱 → 신고필증 OCR → ㉟/㊶ 수량 검증
   
   [검증 실패] → 오류 메시지 표시 → 업로드 페이지로 돌아가기
   [검증 통과] → 코드 매칭 실행
   ↓

5. 매칭 검토 페이지
   ↓
   [모두 자동 확정] → 자동으로 다음 단계
   [확인요 항목 있음] → 담당자가 후보 중 1클릭 확정 → "결과 생성" 버튼
   ↓

6. 안분·환율 결과 페이지
   ↓ 부대비용 안분 + 환율 적용 결과 미리보기 확인 → "다운로드" 버튼

7. 결과 다운로드 페이지
   ↓ 이카운트 양식 엑셀 다운로드 + 검증 리포트 다운로드

8. 완료 → 배치 목록 페이지로 돌아가기
```

## 4. 기능 명세

### MVP 핵심 기능

| ID | 기능명 | 설명 | 우선순위 | 관련 페이지 |
|----|--------|------|----------|------------|
| **F001** | 파일 업로드 | PDF(정산서+신고필증)와 xlsx(입고현황) 드래그&드롭 업로드, 형식·필수 컬럼 유효성 검증 | Must | 업로드 페이지 |
| **F002** | PDF 페이지 분류 | 페이지를 수입정산서(1p) / 세금계산서(skip) / 신고필증(4p~ACE직전) / 보험증권(skip)으로 동적 판정 | Must | 처리 진행 페이지 |
| **F003** | 수입정산서 파싱 | 1페이지 텍스트에서 B/L번호·관세율·부대비용 항목·SUB TOTAL·과세/비과세·환율 구조화 추출 | Must | 처리 진행 페이지 |
| **F004** | 신고필증 Vision OCR | 신고필증 이미지 페이지를 Claude Vision으로 판독해 모델·㉟수량·㊱단가·㊲금액·㊶수량·65환율을 JSON으로 추출 | Must | 처리 진행 페이지 |
| **F005** | 수량 검증 (㉟=㊶) | 신고필증 전 페이지 ㉟수량 합계와 ㊶수량 합계 비교, 불일치 시 진행 차단 및 규정 오류 메시지 반환 | Must | 처리 진행 페이지 |
| **F006** | 입고현황 xlsx 파싱 | 이카운트 24컬럼 양식 파싱, 행별 품목코드·품목명·수량·외화단가·입고일자·통화 추출 | Must | 처리 진행 페이지 |
| **F007** | 코드 매칭 (1순위) | 입고현황 품목코드 앞 10자리와 신고필증 모델번호 일치 여부로 자동 매칭 (정규화 후 비교) | Must | 매칭 검토 페이지 |
| **F008** | AI 의미 매칭 (2순위) | 코드 매칭 미성공 건에 한해 Claude로 영문↔한글 품목명 의미 유사도 점수 산출, 0.90 이상 자동 확정 | Should | 매칭 검토 페이지 |
| **F009** | 수동 확정 UX | "확인요" 상태 품목에 대해 후보를 점수 순으로 제시, 담당자 1클릭 확정 | Must | 매칭 검토 페이지 |
| **F010** | 수량 비교 | 신고수량(㉟) vs 입고수량 비교 후 일치(초록)/불일치(빨강)/확인요(노랑) 상태 배지 표시 | Must | 매칭 검토 페이지 |
| **F011** | 외화단가 역산 보정 | 행별 수량 비중으로 외화단가 배분, Σ(수량×단가)=신고필증 품목 총액 검증 | Must | 안분·환율 결과 페이지 |
| **F012** | 환율 자동 연동 | 입고일 기준 하나은행 최초 고시 환율 조회, 실패 시 서울외국환중개소 폴백, 모두 실패 시 수동 입력 허용 | Must | 안분·환율 결과 페이지 |
| **F013** | 부대비용 안분 | 수량 일치→신고수량 기준, 불일치→입고수량 기준으로 안분 후 운반비·수수료·기타 칸에 분류 기입 | Must | 안분·환율 결과 페이지 |
| **F014** | 원화 환산 | 단가(원화)=외화단가×환율, 공급가액=수량×단가/1.1, 부가세=공급가액×10%, 통화코드 자동 | Must | 안분·환율 결과 페이지 |
| **F015** | 이카운트 양식 생성 | 24컬럼 순서를 그대로 유지한 채 자동 기입 완료된 xlsx 파일 생성 및 다운로드 | Must | 결과 다운로드 페이지 |
| **F016** | 검증 리포트 | ㉟=㊶ 검증·단가 역산 검증·안분 합계 검증 결과, 매칭 상태 요약, OCR 저신뢰 플래그 포함 리포트 | Must | 결과 다운로드 페이지 |

## 5. 메뉴 구조

```
수입정산 매니저 내비게이션

[비로그인 상태]
├── 로그인 - F020
└── 회원가입 - F020

[로그인 후 - 헤더]
├── 홈 (배치 목록)
│   └── 기능: F021 (정산 처리 이력 목록)
```

### F010 — 수량 비교

**비교**: 매칭된 쌍에서 `qty_35`(신고수량) vs 입고현황 `수량`(입고수량) 비교

| 상태 | 조건 | 배지 색상 | 안분 기준 |
|------|------|-----------|-----------|
| 일치 | 신고수량 = 입고수량 | 초록 | 신고수량 기준 안분 |
| 불일치 | 신고수량 ≠ 입고수량 | 빨강 | 입고수량 기준 안분 + "입고수량 기준 안분" 표시 |
| 확인요 | 매칭 미확정 | 노랑 | 확정 후 결정 |

### F011 — 외화단가 역산 보정

**목적**: 같은 모델의 색상·사이즈별 행이 여러 개일 때, 총 외화금액 = Σ(수량 × 단가)가 신고필증 품목 총액과 일치하도록 행별 단가를 역산 보정

**수식**:
```
행별 외화단가(보정) = 신고필증 품목 총액 × (행 수량 / 품목 전체 수량 합계)
                     ÷ 행 수량
```

**검증**: `Σ(행 수량 × 보정 단가) == 신고필증 품목 총액` → 불일치 시 플래그 + 재확인 요청

---

### F012 — 환율 자동 연동

**조회 모듈 흐름**:
```
getFxRate(date, currency)
  1) fx_rate_cache 조회 (date, currency) → 캐시 히트 시 즉시 반환
  2) 캐시 미스 → 1차: 하나은행 기간별 환율조회 (최초 고시, 매매기준율)
  3) 1차 실패 → 2차: 서울외국환중개소 기간별 매매기준율
  4) 결과를 fx_rate_cache에 저장 (source, fetched_at 포함)
  5) 두 소스 모두 실패 → 오류 + 수동 입력 오버라이드
```

**통화코드 자동 결정**: USD → `00001`, CNY → `00002`

**영업일 처리**: 비영업일(주말·공휴일) 입고 시 직전 영업일 고시 적용

**수동 오버라이드**: 입력 시 source를 `manual`로 기록, 안분·환율 결과 페이지에 "수동 입력" 표시

---

### F013 — 부대비용 안분

**A) 수량 일치 → 신고수량 기준**:
```
품목 안분액 = 부대비용 총액 × (품목 신고수량 ÷ 전체 수량 합계)
```

**B) 수량 불일치 → 입고수량 기준**:
```
품목 안분액 = 부대비용 총액 × (품목 입고수량 ÷ 전체 입고수량 합계)
```
해당 품목 행에 "입고수량 기준 안분" 표시, 수량 차이를 비고란에 기록.

**칸별 안분 적용**:

| 이카운트 칸 | 정산서 항목 | 안분 방식 |
|---|---|---|
| 운반비 | 통관수수료 + 업무운임 SUB TOTAL | 수량 비중 안분 |
| 수수료 | 통관부가세 | 수량 비중 안분 |
| 기타 | 관세 | 수량 비중 안분 |

**반올림 처리**: 원 단위 반올림 후 잔차는 최대 수량 품목에 가산

**검증**: `Σ(품목 안분액) == 정산서 해당 칸 총액` → 불일치 시 오류 표시

---

### F014 — 원화 환산

```
단가(원화)  = 외화단가(보정) × 환율
공급가액    = 수량 × 단가(원화) / 1.1
부가세      = 공급가액 × 10%
외화금액    = 수량 × 외화단가(보정)
통화(코드)  = USD → 00001 / CNY → 00002
```

## 10. 검증·수용 기준 (Acceptance Criteria)

### AC-01: 수량 검증 (㉟=㊶)

```
GIVEN 신고필증 4페이지가 정상 OCR 처리된 배치가 있을 때
WHEN 각 페이지의 ㉟ 수량을 합산하고 ㊶ 수량을 합산할 때
THEN 두 합계가 일치하면 다음 단계로 진행
AND  두 합계가 불일치하면
     "수입신고필증의 35번 수량 합계와 41번 수량 합계가 일치하지 않습니다." 메시지를 표시
     이후 단계로 진행 불가
     validation_log에 type=qty3541, passed=false 기록
```

### AC-02: 코드 매칭 정확성

```
GIVEN 입고현황 품목코드 앞 10자리가 신고필증 모델번호와 동일한 배치가 있을 때
WHEN 코드 매칭을 실행할 때
THEN 해당 품목은 자동으로 "일치" 상태로 매칭
AND  item_match 레코드에 method=code 기록
AND  B/L번호가 해당 행에 자동 연결
```

### AC-03: 부대비용 안분 합계 검증

```
GIVEN 3개 품목의 부대비용 안분 계산이 완료된 배치가 있을 때
WHEN 각 칸(운반비·수수료·기타)의 품목별 안분액을 합산할 때
THEN 각 칸의 합계가 정산서의 해당 칸 총액과 정확히 일치 (원 단위 반올림 잔차는 최대 수량 품목에 가산)
AND  불일치 시 validation_log에 type=alloc_*, passed=false 기록
```

### AC-04: 환율 자동 연동

```
GIVEN 입고일자가 2026-05-19인 USD 품목이 있을 때
WHEN 환율 조회를 실행할 때
THEN 하나은행 최초 고시 매매기준율을 우선 조회
AND  조회 성공 시 fx_rate_cache에 source=hana로 저장
AND  하나은행 조회 실패 시 서울외국환중개소로 폴백
AND  두 소스 모두 실패 시 수동 입력 필드 표시 (source=manual 기록)
```

### AC-05: 이카운트 양식 완성도

```
GIVEN 모든 품목 매칭·안분·환산이 완료된 배치가 있을 때
WHEN 이카운트 양식 다운로드를 실행할 때
THEN 24컬럼 순서가 원본 이카운트 양식과 동일한 xlsx 파일 생성
AND  영업팀 입력 컬럼은 원본 값 보존
AND  자동 기입 컬럼(환율·단가·공급가액·부가세·운반비·수수료·기타)은 계산값으로 채움
AND  BL번호는 적요 컬럼에 자동 기입
```

### AC-06: OCR 저신뢰 플래그

```
GIVEN 신고필증 특정 페이지의 confidence가 0.85 미만일 때
WHEN 처리 진행 페이지를 표시할 때
THEN 해당 페이지에 "OCR 저신뢰" 경고 배지 표시
AND  원본 이미지 링크 제공
AND  처리는 중단하지 않고 계속 진행 (담당자 확인 권고)
```

## 14. 가정 및 오픈 이슈

### 가정

| # | 가정 내용 |
|---|-----------|
| G1 | 최종정산서 PDF는 항상 1페이지가 수입정산서(텍스트 레이어), 4페이지부터 신고필증(이미지)임 |
| G2 | 세금계산서는 항상 2~3페이지에 위치하고 처리 불필요 |
| G3 | ACE 보험증권 시작 문구(`"ACE American Fire and Marine Insurance Company Korea, A Chubb Company"`)는 신고필증 이후 최초 등장 |
| G4 | 이카운트 xlsx 양식의 컬럼 순서와 헤더명은 고정 (24컬럼) |
| G5 | 입고현황 품목코드 앞 10자리가 신고필증 모델번호와 대응함 |
| G6 | 처리 통화는 USD와 CNY 두 종류로 한정 |
| G7 | 하나은행·서울외국환중개소 웹사이트에서 날짜별 환율 파싱이 가능한 구조임 |

### 오픈 이슈

| # | 이슈 | 영향도 | 대응 방안 |
|---|------|--------|-----------|
| OI-1 | **환율 외부 소스 연동 불안정성**: 하나은행·서울외환 웹사이트 구조가 변경되면 파싱 실패 | 높음 | 어댑터 패턴 분리 + 수동 오버라이드 안전장치 + 정기 모니터링 |
| OI-2 | **OCR 신뢰도 임계값 0.85 적정성**: 실측 신고필증 배치로 검증 필요 | 중간 | 초기 운영 후 실측 confidence 분포 확인하여 조정 |
| OI-3 | **정산서 1페이지 파싱 내 CID 폰트 한글 깨짐**: PyMuPDF 기반 텍스트 추출 시 한글이 깨질 수 있음 | 높음 | 텍스트 레이어 우선, 깨짐 감지 시 이미지 렌더링 → Claude Vision 폴백 경로 설계 |
| OI-4 | **신고필증 구조 다양성**: UNI-PASS 서식 외 다른 서식 신고필증 처리 가능 여부 미확인 | 중간 | MVP는 UNI-PASS 서식 기준, 미지원 서식은 "OCR 저신뢰" 처리 후 수동 확인 |
| OI-5 | **Claude API 비용·속도**: 신고필증 페이지 수 증가 시 OCR 비용 및 처리 시간 증가 | 중간 | 페이지 병렬 처리, OCR 전용 모델 최적화, 필요 시 캐싱 |
